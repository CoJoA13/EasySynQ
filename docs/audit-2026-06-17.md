## Executive summary

- **Structurally sound, but the status/index surfaces have drifted badly.** The design-spec prose
  (docs 01–18 bodies, decisions register, the OpenAPI contract) is largely accurate. The damage is
  concentrated in **front-door status surfaces** (`README.md`, `docs/16-roadmap.md`, `docs/17`
  resolution status) and **as-built notes** embedded in otherwise-correct schema/authz docs.
- **Single most important issue:** `README.md` is ~25 migrations and two whole families behind reality
  (v1 "in progress" at Records + Ingestion S-ing-1; migration head `0029` vs actual `0054`; Decisions
  Register cited R1–R37 vs actual R1–R46). A new reader's first impression of project state is wrong.
- **Dominant pattern — "built-but-documented-as-deferred":** ≥8 findings assert shipped work is
  unbuilt/TODO/next. The **Improvement Initiatives** family (R46) and **Leadership routing**
  (S-leadership-1) are the most-missed.
- **Cross-doc drift on two facts:** the Decisions Register range (canonical **R1–R46** in
  CLAUDE.md/doc 00; stale at **R37** in README/doc 14/doc 18, **R40** in diff-critic.md) and the
  migration head (**0054** canonical; 0029/0045/0050 in README/roadmap). The `signed_object_type` /
  `workflow_subject_type` enums in doc 14 and doc 10 lag code + contract.
- **Tooling commands diverge from CI:** `.claude/commands/{check-web,check-api,pr}.md` omit `npm test`
  and/or run bare `uv run pytest` (which collects Docker-only integration tests) — falsely-green local gates.

## Detailed P1 / P2 findings

### P1-1 — docs/03-architecture-and-stack.md §6.2 (engine FSM diagram)
The §6.2 mermaid diagram is the **engine** (7-state) FSM — it contains `UnderRevision`, so it is *not*
the simplified 5-state UI view — yet it uses `Released` (non-canonical; R1 canon is `Effective`) in four
spots, omits `Superseded` entirely, and its note ("Only Released versions are written to the … mirror")
contradicts doc 04 §3.1 L144 ("Only **Effective** versions are written"). No R1/seven-state note present.
**Fix:** rename `Released`→`Effective` (×4), add `Effective→Superseded`, fix the mirror note, add an R1 reference.

### P1-2 — docs/14-data-model.md L461
> **`improvement_initiative` is deferred/unbuilt in v1** (improvement-opportunity inputs ship as honest gap rows).

The schema source-of-truth doc tells a reader the model is not built, but the Improvement Initiatives
family shipped end-to-end (`improvement_initiative.py` + `improvement_initiative_stage_event.py`,
migration 0052; auth 0053). The clause is headed "As-built (S-mr-1)" but describes a *different* family
with no historical marker. **Fix:** replace with an as-built note pointing to the own-table model + doc 15 §8.12b.

### P1-3 — docs/16-roadmap.md §4
The living §4 status block stops at S-mr-1. Improvement Initiatives (R46, migs 0052/0053) and Leadership
routing (S-leadership-1, mig 0054, #190/#192) are absent; every post-MR slice shipped (S-mr-2/-3/-pack,
KPI charts, PDCA Home, drift D1–D4, acks UI). Highest migration cited is 0050 vs actual head 0054.
**Fix:** refresh §4 to as-built (or convert to a dated snapshot + defer per-slice status to slice-history + CLAUDE.md); add the two missing families; head→0054.

### P1-4 — README.md (Status; L82; L106)

the feature-complete web track; (2) migration head `0029` is 25 behind `0054`; (3) Register cited R1–R37
vs the 46 contiguous headings. Contradicts CLAUDE.md and doc 00 (both R1–R46).
**Fix:** rewrite Status to completed families + head 0054 (mirror CLAUDE.md or point to slice-history); R1–R37→R1–R46.

### P1-5 — docs/slice-history.md (Leadership Routing section)

not-yet-built "next slice" — but that FE shipped as PR #192 / commit `9681b6d`, the literal HEAD of main

import-baseline ungated, the DCR-implement integration test) remain accurate.

### P1-6 — docs/07-authorization-model.md (catalog)

non-sig/non-SoD) and grants them (QMS Owner / Process Owner read+manage; Internal Auditor read). The live

were correctly added as the additive-key precedent; `improvement.*` was missed.
**Fix:** add the two keys to §3 + the §3.10 summary, reflect the grants in §4.2/§9.1, update "stays 100"→102.
