### CR-1. The check-in ↔ records WORM/blob data-loss chain
_Same root cause, two code sites (each needs a fix), surfaced independently by the vault, schema-design, and records lenses._ · **CONFIRMED**

**Site — `apps/api/src/easysynq_api/services/records/repository.py:431`.** The destroy-time liveness guard `blob_needed_by_other_live_record` only checks other records' `evidence_blob` attachments; it never checks `document_version.source_blob_sha256`/`rendition_blob_sha256` (both RESTRICT FKs onto `blob.sha256`, document_version.py:78/82). This matters because the vault check-in path creates the cross-reference: `services/vault/service.py` `initiate_upload` (line 444-447) dedups on ANY existing blob row, and `checkin` (lines 502-534) reuses an existing blob row with NO bucket check — so checking in a file whose bytes were previously captured as record evidence yields a `document_version` whose `source_blob_sha256` points at the records-bucket blob. (The reverse direction IS guarded: `_attach_evidence` 423s on a blob vaulted outside the records bucket, service.py:358-376 — the check-in side lacks the symmetric guard.) When that record is later destroyed (manual PATCH, the auto-sweep for a review_required=false DESTROY policy, or the R27 hatch), `_purge_record_evidence` (disposition.py:92-97) sees no other live RECORD, physically purges the object — destroying the controlled document version's source-of-truth bytes (D2 break) — and then `delete_blob_and_links` hits the document_version RESTRICT FK: an IntegrityError, not the refuse-with-reason contract. The txn rolls back, restoring the blob row over already-deleted bytes → blob-row-without-bytes. The backup RUN still SUCCEEDS (build_durable_backup/_capture_and_dump manifests only blob METADATA, never fetching bytes), silently producing an archive with a dangling reference; the NoSuchKey surfaces later at _copy_blobs during a restore drill, retained-backup verification, or an actual restore — so backups keep appearing successful while being UNRESTORABLE (the S-rec-2 invariant class; worse than a loud crash). In the Beat sweep the IntegrityError is not in the `(ClientError, BotoCoreError)` catch (disposition.py:549), so the whole sweep crashes, loses the run's other work, and re-crashes every day (purge_object is idempotent, the FK conflict is permanent). Secondary: even before any destroy, `packs/build._gather_version_files` (build.py:91-93) fetches the source hard-coded from `s3_bucket_documents`, so a pack pinning such a version fails NoSuchKey → pack FAILED.

_Failure scenario._ 1) Priya captures an EVIDENCE record attaching report.pdf (sha X) under a DESTROY/review_required=false policy → blob(X, bucket=records). 2) Later she checks the identical file in as a controlled document: init-upload returns dedup=true, checkin reuses blob X → document_version.source_blob_sha256=X; the doc goes Effective. 3) Retention elapses; the nightly sweep auto-disposes the record: purge_object deletes X's bytes (the Effective document's only source bytes — data loss), then DELETE blob X raises IntegrityError (document_version RESTRICT) → the sweep crashes, the DB rolls back leaving blob row X pointing at deleted bytes → subsequent backups still SUCCEED (metadata-only manifest) yet are silently unrestorable (the NoSuchKey surfaces at restore/verify in _copy_blobs), the mirror rebuild of that document fails, and every subsequent sweep crash-loops on the same record.

_Fix._ Add a document_version leg to blob_needed_by_other_live_record (refuse-with-reason, mirroring the WORM-lock refusal) AND close the root cause: make vault checkin/checkin_* fail-closed (423) when the existing blob row's bucket is not s3_bucket_documents — the exact symmetric guard _attach_evidence already applies.

**Site — `apps/api/src/easysynq_api/services/vault/service.py:501`.** checkin() dedups on the global content-addressed blob PK: `blob = await repository.get_blob(session, sha256); if blob is None: promoted = await storage.finalize_worm(...)`. When a blob row for that sha ALREADY exists, checkin skips finalize_worm and all WORM checks and mints the document_version against whatever row is there — regardless of its bucket or worm_locked. The two sibling paths explicitly guard exactly this: ingestion commit (services/ingestion/commit.py:326-327 `elif blob.bucket != settings.s3_bucket_documents: raise _ItemCommitError("source_bytes_in_foreign_bucket")` — "the symmetric cross-kind sha collision") and records _attach_evidence (services/records/service.py:358-377, 423 worm_required on a blob "already vaulted outside the records bucket"). The vault checkin (and the six checkin_* register/form variants which share the shape, lines 758, 856, 955, 1054, 1154, 1255) have no such guard, and init_upload (line 444-449) tells the client `dedup: true` for a blob in ANY bucket, so the client never even uploads.

_Failure scenario._ (a) Non-WORM source on the common path: a user downloads the Effective controlled-copy PDF via GET /documents/{id}/download (presigned from the renditions bucket; byte-identical to the `_cache_rendition` blob), then checks that PDF in as a new version of any document. init-upload answers dedup:true; checkin finds the existing renditions-bucket blob row (worm_locked=False, non-WORM bucket) and creates the immutable document_version pointing at it — the load-bearing "WORM-before-version-complete" invariant is silently bypassed for a controlled version's source bytes. (b) Data-loss chain: bytes are first captured as record evidence (blob row bucket=records), later checked in as a document source (checkin reuses that row). When the record's retention lapses, the disposition sweep sees no OTHER live record needs the blob, physically destroys the bytes via purge_object, then delete_blob_and_links hits the document_version FK RESTRICT → IntegrityError; the txn rolls back the row deletes but the object bytes are already gone. Result: an (even Effective) document's source bytes are physically destroyed, the blob row survives pointing at nothing (blob-row-iff-bytes broken → every backup/restore drill crashes NoSuchKey), and the mirror rebuild raises on fetch_bytes every run.

_Fix._ Mirror the ingestion-commit guard in checkin()/checkin_*: when a blob row exists, require `blob.bucket == s3_bucket_documents and blob.worm_locked` (else 423/422), and make init_upload return dedup:true only for a documents-bucket WORM blob. ⚠ Guarding only the pre-existing-row branch is NOT enough: on a CONCURRENT first insert of a new sha (checkin + record-evidence both see no row and both `INSERT … ON CONFLICT DO NOTHING`), the loser silently keeps its stale local view and would reference the winner's wrong-bucket row. So each path must RELOAD the authoritative blob row AFTER the conflict-tolerant insert and re-validate its bucket/worm_locked before creating the reference (fail-closed on mismatch).

### CR-2. Chain-linker links audit rows out of id order under normal concurrency, permanently breaking chain verification
**`apps/api/src/easysynq_api/services/audit/linker.py:68`** · Workers / backup / Celery · **CONFIRMED** · cross-confirmed by: read-surfaces, workers

link_org_chain selects ALL visible rows WHERE chained_at IS NULL (no guard against still-in-flight transactions holding lower, already-allocated ids), and _seed_prev_hash seeds from the max-chained-id row. A row INSERTed (flushed) by a still-open transaction is invisible to the linker's READ COMMITTED SELECT; if a higher-id row from another transaction commits and a linker tick (Beat, every 30s) runs before the first transaction commits, the higher-id row is linked first. The next tick links the lower-id row with prev_hash = hash(higher-id row). verify_chain (services/audit/verify.py:55-98) walks strictly by id ASC and requires prev_hash[n] == row_hash[n-1], so both rows are flagged 'prev_hash mismatch (chain reorder/deletion)' — permanently: chained rows are never re-linked, and prev_hash/row_hash UPDATE is REVOKEd from all roles except easysynq_linker (which only processes chained_at IS NULL). Mid-transaction flushes of audit rows with long open-tail windows are real in this codebase: sweep_acks mints a whole org's DOC_ACK batch with per-instance flushes (services/ack/sweep.py:230 via engine.py:335, AuditEvent adds at engine.py:106) and ONE commit at sweep end (sweep.py:277) — pubsub.py:27-29 explicitly calls this caller txn 'long'; sweep_reviews and the SERIALIZABLE _cutover have the same shape.

**Failure scenario.** Daily ack sweep starts minting DOC_ACK obligations for 50 users: instantiate() flushes, INSERTing TASK_CREATED audit rows ids 1000-1049 in the open sweep txn. Mid-sweep, a user releases a document; its RELEASED audit row gets id 1050 and commits immediately. The 30-second easysynq.audit.chain_link tick fires, sees only id 1050 unchained, links it with prev=hash(999). The ack sweep commits; the next tick links 1000-1049 with 1000.prev=hash(1050). The nightly easysynq.audit.verify_chain (and every future GET /audit-events/verify-chain, and the S11 restore re-verify of any backup of this DB) reports permanent 'prev_hash mismatch (chain reorder/deletion)' breaks at ids 1000 and 1050 — an unrepairable false tamper alarm on the load-bearing WORM audit chain, indistinguishable from real tampering.

**Fix.** Make the linker chain only a PROVEN contiguous safe `id`-prefix. This is a non-trivial concurrency design — the fix must satisfy ALL of: (a) `audit_event.id` is one GLOBAL sequence but chains are PER-ORG, so a 'missing' id in org A is normally another org's committed row — treat any visible committed cross-org row as immediately safe; (b) a globally-absent id is NOT provably rolled back just because a grace window elapsed — a long-open audit-writing txn can commit its already-allocated lower id after any fixed grace, which would reproduce the reorder — so advancing past a gap needs a real PROOF every txn that could own it has ended (a durable allocation/commit registry, or a tracked transaction/snapshot horizon), not elapsed time; (c) STOP at the first such unproven gap. ⚠ Do NOT bound by `xmin`/transaction age alone (xids don't track `id` order) and do NOT make verify walk `(chained_at, id)` order (that defeats the `id`-ordered reorder detection docs/12 §4.3 requires). Keep verify's strict `id`-order walk; the safe-prefix proof lives in the linker.

### CR-3. _is_system_tier ignores ABAC predicates: an expired system-tier permission.grant override permanently elevates a content-tier grantor
**`apps/api/src/easysynq_api/services/authz/pep.py:194`** · Authorization (PDP/PEP) · **CONFIRMED**

The R35 two-tier guard's tier classification scans raw grants with no predicate evaluation: `return any(g.effect is Effect.ALLOW and not (g.predicates or {}).get("content_only") for g in grants)`. gather_grants folds override valid_from/valid_until into g.predicates (repository.py:73-76), but _is_system_tier never runs _context_predicates_pass or authorize(), so an EXPIRED (valid_until in the past), not-yet-valid (valid_from in the future), or ip_allow-restricted non-content_only permission.grant override still classifies the caller as system-tier. Expired overrides are never garbage-collected (no sweep deletes PermissionOverride rows), so the elevation is permanent. The route dependency require("permission.grant") is satisfied by the caller's separate live content-tier grant (e.g. the seeded QMS Owner role's content_only grant), so the lapsed override is never re-validated anywhere.

**Failure scenario.** Admin grants the QMS Owner a time-boxed deputy-admin window: POST /users/{qms_owner}/overrides {permission_key: "permission.grant", effect: ALLOW, valid_until: <+7d>} (system-tier, no content_only). After day 7 the override no longer matches in the PDP, but assert_can_grant/assert_can_assign_role still call _is_system_tier -> True (the expired ALLOW counts). The QMS Owner — whose live authority is content-tier only — can now grant backup.restore/user.create/config.update overrides or assign the System Administrator role to anyone, indefinitely, crossing the R35 admin-only boundary (D-rule: ADMIN sits outside the QMS and vice versa).

**Fix.** In _is_system_tier, decide the question correctly: does an EFFECTIVE non-`content_only` SYSTEM ALLOW survive the FULL deny/scope/predicate evaluation against `ResourceContext.system()`? Two traps make the naive versions wrong: (a) `_context_predicates_pass` alone ignores SCOPE — `OverrideCreate` accepts any scope level without enforcing `Permission.finest_scope`, so a non-`content_only` PROCESS-scoped `permission.grant` override passes a predicates-only test yet can't authorize SYSTEM; (b) inspecting the single 'winning' grant is nondeterministic — a user holding BOTH System Administrator and QMS Owner has two SYSTEM `permission.grant` grants (one without `content_only`, one with) that the PDP ranks identically and `gather_grants` returns unordered, so the tie-winner could be the content-tier one and misclassify a real admin. So evaluate whether ANY non-content_only SYSTEM grant survives deny+scope+predicates (not the tie winner), e.g. filter grants to non-`content_only`, run them through the full PDP against `system()`, and treat a surviving ALLOW as system-tier. (If you reuse `authorize`, note `Decision` carries only a provenance string today, not the marker.)
