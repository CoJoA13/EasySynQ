> **The thesis.** A Management Review *inverts* the meeting-vs-evidence tension. The convened review's **minutes** — the auto-compiled 9.3.2 inputs *as-of*, plus the 9.3.3 decisions/outputs — are a **controlled Document**: versioned, signed off by management, clause-mapped to **9.3**, **released** as the immutable filed evidence (the 9.3.3 "retained documented information"). The **post-release action loop** — tracking each output to closure — is **operational state** on a mutable satellite column, never a controlled revision. The MR row is a shared-PK subtype of `documented_information` (`kind=DOCUMENT`, type `MR`); the compiled inputs + decisions **freeze into the version snapshot at submit** (the S-obj-3 recipe verbatim); **release flips the 9.3 ★** with zero checklist code; outputs **spawn `MR_ACTION` tasks**; the review **closes only when its output actions close** (the `_audit_close_gate` pattern); a **Beat cadence sweep** mints the next Scheduled review automatically.

## s0 · Owner decisions (this session, 2026-06-12)

The brainstorm was preceded by a nine-reader source-verification sweep (every claim below checked against code, not slice-history). The F-numbers are the owner's decision frame.

1. **F2 verdict — only a released `kind=DOCUMENT` flips the 9.3 ★ node.** The checklist's COVERED predicate is *literally* `current_effective_version_id IS NOT NULL` on a row mapped to the **exact** clause `9.3` (no subtree rollup — 9.3.1/9.3.2/9.3.3 are non-★ and do **not** count), and that pointer is set only by the document release cutover (`lifecycle.py:513`) / ingestion baseline (`commit.py:355`). A `kind=RECORD` aggregate sets `current_state=Effective` but never a version pointer → it would sit at **PARTIAL forever** (and PARTIAL "looks started forever" — worse than a clean GAP→COVERED). *Source: `checklist.py:13-14,39-45,84-122`; proven by `test_objective_lifecycle.py:127-169` (OBJ flips 6.2 only after release sets the pointer).*

2. **F1 — DOCUMENT head (the OBJ recipe).** `management_review` is a shared-PK subtype of `documented_information` (`kind=DOCUMENT`, `document_type='MR'`), exactly like `quality_objective`. It auto-maps to clause 9.3 at create and rides the unchanged submit→approve→release machinery → the ★ flips on release with **zero `checklist.py` change**. The doc-10 §7 operational states map onto the authoring FSM (Scheduled→Draft, Preparing→Draft, Convened→submit/InReview, Decided→Approved, *filed*→Effective) with a mutable `close_state` carrying the post-release ActionsTracked→Closed tail.
   - *Grounding:* the OBJ/S-obj-3 path is the proven, lowest-risk mechanism; release ALSO mints `signature_event(meaning=approval/release)` for free (s5/F7).
   - *Rejected:* **(A) `kind=RECORD` head + widen `coverage_status()`** — doc-14 §9's literal prescription, but it changes a **load-bearing, generic** checklist predicate (a per-family `OR (MGMT_REVIEW record AND cycle=Closed)`), couples the query to the MR satellite, and risks a flip-on-create trap (a captured record lands Effective instantly). **(B) Hybrid** (RECORD ops + a DOCUMENT minted at close) — most faithful to *both* docs but two coupled aggregates, the heaviest slice. DOCUMENT wins on risk + reuse + the free signed sign-off.
   - *Register obligation:* the new **R45** entry records the **deliberate deviation** from doc-14 §9's RECORD classification (sanctioned — the MR docs are "rich but unbuilt and not register-locked"; opening the family is a register call).

5. **F3 — inputs: compile under the review-owner's identity, fail-closed per source, freeze at submit.** A `compile-inputs` call (re-runnable in Draft) attempts each of the six live reads **gated on the review *owner's* grants** (the MR document's `owner_user_id`), **not the caller's** — so the frozen 9.3 evidence is **deterministic**: the same review compiles the same pack regardless of which authorized preparer clicked it. A source the owner can't read (or that has no backend) becomes a **named "not available" gap row**, never a 403 of the whole compile. (a) prior-review actions resolves once a second review exists.

7. **F5 / F5b — `MR_ACTION` tasks at release; the close gate requires spawned work CLOSED.** An output with an owner spawns one `MR_ACTION` task at release (assignee = the output owner, the periodic-review `context_users` seam; `task_type` is data-driven in `_materialize_stage` — **no engine edit, no migration**). The review-close gate **mirrors `_audit_close_gate`**: 409 the close while any action-output's `MR_ACTION` task is not DONE (None = fail-closed). **Defer** CAPA un-reserve (the `review_output.spawned_capa_id` seam ships reserved-null; un-reserving = the 4-site change in slice-2) and the DCR `mgmt_review` link (its `source_link_id` has no FK → would dangle).

The Home "next review in N days" widget lands in **S-mr-2**.

**Accepted defaults (no pushback):** sequencing backend → UI → (Home widget in the UI slice); MR documents reuse `create_document`/checkin/release (no new lifecycle); `improvement_initiative` deferred entirely (a reserved-null `review_output.spawned_initiative_id` seam, no table); imported legacy minutes stay `kind=RECORD` type `MGMT_REVIEW` (ingestion's existing classification — they are *evidence*, do not flip the ★, and coexist with the new authored `MR` documents); the new register entry is **R45**.

## s3 · The input compiler (`services/mgmt_review/compile.py` + a pure `domain/mgmt_review/inputs.py`)

`compile_inputs(session, review, caller)` (Draft only) attempts each sourced 9.3.2 read **gated on the review owner's grants** (the MR document's `owner_user_id`, PDP-checked per key — `caller` only authorizes *triggering* the compile via `mgmtReview.record_outputs`) and writes one `review_input` row per `ReviewInputType`:

- **`OBJECTIVES_STATUS` (c2)** ← the objectives scorecard (`objective.read`) → `{total, on_target, by_rag}`.
- **`AUDIT_RESULTS` (c6)** ← audits read (`audit.read`) → `{open, closed, coverage}`.
- **`NONCONFORMITIES_CAPA` (c4)** ← NCRs (`ncr.read`) + complaints (`record.read`) + CAPAs (`capa.read`) → `{open_ncrs, open_capas, by_close_state}`.
- **`MONITORING_RESULTS` (c5)** ← KPI readings (`kpi.read`) → `{readings, objectives_measured}`.
- **`PROCESS_PERFORMANCE` (c3)** ← the compliance checklist + overdue rollup (`report.compliance_checklist.read`) + drift status (`drift.read`) → `{star_coverage, overdue_reviews, integrity}`.
- **`PRIOR_ACTIONS` (a)** ← the previous released MR's `review_output` rows + their `spawned_task_id`/`spawned_capa_id` terminal states (gap row until a second review exists).
- **Gap rows** (`available=false`, a fixed neutral `reason`): `CONTEXT_CHANGES` (b), `CUSTOMER_SATISFACTION` (c1), `SUPPLIER_PERFORMANCE` (c7), `RESOURCE_ADEQUACY` (d), `RISK_OPPORTUNITY_ACTIONS` (e), `IMPROVEMENT_OPPORTUNITIES` (f).

**Fail-closed per source (F3):** each sourced read is attempted inside a guard; the **owner** lacking that key (PDP-checked) writes an `available=false` row with `reason="not available (insufficient access)"` — **never** a 403 of the whole compile. (In practice the QMS-Owner-level review owner holds the union; a thinner owner simply yields more gap rows — but deterministically, independent of the clicker.) The PDP check keys on the owner's gathered grants; the underlying service reads (`list_audits(session, org_id)`, the scorecard service, …) are not re-impersonated — the authz decision is separated from the data read. The summaries are **as-of snapshots** (counts/RAG/top items), not live queries retained for read-time re-grading. Audit `MGMT_REVIEW_INPUTS_COMPILED`.

The pure `domain/mgmt_review/inputs.py` owns the read-shape→summary projection (total, deterministic, unit-tested), keeping `compile.py` to orchestration + authz guards.

## s4 · Lifecycle & the freeze

1. **Scheduled** — the cadence sweep (s6) or `POST /mgmt-reviews` mints a **Draft** `MR` document (`kind=DOCUMENT`, type `MR`, owner = the convening QM) + the `management_review` satellite in one txn via `create_document`; **auto-creates a `clause_mapping` to the 9.3 leaf** (mirroring `objectives/service.py:137-156`, so the submit gate's ≥1-mapping requirement and the ★ both resolve). The sweep also mints an `MR_INPUT` task. `close_state` NULL.
2. **Preparing** (Draft) — `POST /mgmt-reviews/{id}/compile-inputs` (re-runnable, replaces the working `review_input` set); `POST/PATCH/DELETE /mgmt-reviews/{id}/outputs` author decisions/actions (`mgmtReview.record_outputs`). Set `review_date`/`attendees`/`period_label`.
3. **Convened** — `POST /mgmt-reviews/{id}/submit-review` (`mgmtReview.record_outputs`) → Draft→InReview. The minutes **freeze**: `checkin_mgmt_review_minutes` builds `{period_label, review_date, attendees, inputs:[…source_ref…], outputs:[…decision content…], compiled_at}` (every leaf a JSON-safe primitive — dates `.isoformat()`, Decimals `str()`) → `rfc8785.dumps` (hashed **bare, no preamble**) → `put_staging_bytes(application/json)` → `finalize_worm` (the version source blob) → the **same dict** into `metadata_snapshot.mgmt_review_minutes` via the `_snapshot` one-kwarg fold. After submit the inputs/decision-content are immutable (the snapshot is authority; the `review_input`/`review_output` rows are frozen by the doc leaving Draft). Submit rides the OBJ `submit_objective_for_review` shape: the freeze (a sub-step that flushes), the generic `submit_review` (Draft→InReview), `instantiate_approval` (a **DOCUMENT**-subject approval instance — the GET approval queries `WorkflowSubjectType.DOCUMENT`), one commit.
4. **Decided** — `document.approve` (SoD: submitter≠approver) → `signature_event(meaning=approval)`.
5. **Effective / filed** — `POST /mgmt-reviews/{id}/release` (`document.release` + SoD) → the standard SERIALIZABLE cutover sets `current_effective_version_id` → **9.3 ★ COVERED** + `signature_event(meaning=release)`. **On release:** for each `review_output` of type `ACTION` (owner present), mint an `MR_ACTION` task (assignee = `owner_user_id`, `task_type=MR_ACTION`, `due_at` = `due_date` at org-midnight) and set `spawned_task_id`; set `close_state=ActionsTracked`; audit `MGMT_REVIEW_ACTION_SPAWNED` per task.
6. **ActionsTracked → Closed** — each `MR_ACTION` is decided through the standard `/tasks` inbox. `POST /mgmt-reviews/{id}/close` (`mgmtReview.record_outputs`) runs the **close gate** (s5) → on pass, `close_state=Closed` + `closed_at` + audit `MGMT_REVIEW_CLOSED`.

The pre-release "cycle" the UI shows is **derived** (`Draft→Preparing`, `InReview→Convened`, `Approved→Decided`, `Effective`+`close_state`) — no redundant stored state, no dual source of truth.

## s5 · Outputs → work & the close gate

- **Spawn (F5):** `MR_ACTION` tasks are minted **at release** from `ACTION` outputs (s4.5). ⚠ **A Task has no standalone form** — `Task.instance_id` is NOT-NULL → every task hangs off a `WorkflowInstance`, which needs a `definition_id` → so the slice seeds a minimal `management_review` `WorkflowDefinition` (the instance is a container; tasks are added via the S5 direct-insert `WorkflowInstance + Task` pattern, **not** the engine's `_materialize_stage`). One `MGMT_REVIEW` instance per review (subject = the MR doc) carries the `MR_INPUT` task (at create/sweep) and the `MR_ACTION` tasks (at release). Each `MR_ACTION` is `Task(type=TaskType.MR_ACTION, assignee_user_id=output.owner, candidate_pool=[str(owner)], due_at=org-midnight(due_date))`. `WorkflowSubjectType.MGMT_REVIEW` + `TaskType.MR_INPUT`/`MR_ACTION` already exist (no `ALTER TYPE`). An output with **no** owner is a `DECISION` (recorded, untracked) and does not spawn.
- **Decide leg (new):** ⚠ the `/tasks/{id}/decision` endpoint dispatches by `instance.subject_type` and has **no `MGMT_REVIEW` branch** — an MR task would fall through to the DOCUMENT path and 422/mis-scope. The slice **adds** a `if instance.subject_type is WorkflowSubjectType.MGMT_REVIEW: return await decide_mr_task(...)` leg (before the fallthrough) → a new service that **404-collapses** non-membership (assignee/`candidate_pool`, never 403), calls `wf_engine.decide(..., _commit=False)` (the task→DONE + idempotency-replay engine), then commits. Mirrors `decide_periodic_review`. The `GET /tasks` LIST already filters `?type=MR_ACTION` (no API change); only the LIST omits `subject_id` (the S-ack lesson — the per-task page reads it from `GET /tasks/{id}`).
- **Close gate (F5b):** pure `output_blocks_close(output_type, owner_present, spawned_task_state)` → True for an `ACTION` output whose `MR_ACTION` task is not `DONE` (None = fail-closed), mirroring `finding_blocks_close`. `_mgmt_review_close_gate` sums blockers and 409s `mgmt_review_close_blocked` with the count + an honest client-readable readiness note. **Spawned work must reach DONE/Closed**, not merely be dispatched.
- **Deferred (slice-2):** CAPA un-reserve (relax the `capa/service.py:369` 422 + invert `test_capa.py:158` + add `review_output` to `RaiseCapaModal` SOURCES + flip its pinning test + the `openapi.yaml` note — the 4-site change; then the gate also reads `spawned_capa_id` CAPA `close_state==Closed`) and the DCR `mgmt_review` link (no FK → would dangle). The `spawned_capa_id`/`spawned_initiative_id` columns ship **reserved-null** so slice-2 needs no migration.

## s6 · Cadence (Beat sweep — `services/mgmt_review/cadence.py` + `tasks/mgmt_review.py`)

`sweep_mgmt_reviews` (daily, mirrors `sweep_reviews`): take `LOCK_MGMT_REVIEW_SWEEP` (session-scoped advisory, single-flight); read `system_config.mgmt_review_cadence_months` (coded default 12); the horizon = the last released MR's `effective_from` + cadence (first review anchored on org go-live / a configured start when no history); **idempotency** — skip if an open (non-Effective, non-closed) MR document already exists; within the lead window, **create** the next Scheduled Draft MR (via `create_document` under the configured MR owner — typically the QMS Owner) + `instantiate` the `management_review` workflow to mint the `MR_INPUT` task; **degrade to a logged no-op** if the definition is mis-seeded. A sweep mints + audits only — it never escalates a task (`decide()` accepts only PENDING).

**Registration (load-bearing):** import `mgmt_review` in `tasks/__init__.py` (`# noqa: F401`); add the `mgmt-review-sweep` Beat entry (`schedule 86400.0`) in `tasks/app.py`; add `tests/unit/test_mgmt_review_task_registration.py` (`'easysynq.mgmt_review.sweep' in app.tasks` + the daily beat-schedule assertion) — or `.delay` publishes to a name no worker handles.

> *Implementation note:* minting a **document** from a sweep is the one bit heavier than `sweep_reviews` (which re-reviews an existing doc). The lighter fallback — mint only an `MR_INPUT` "schedule the review" to-do and let the QM `POST /mgmt-reviews` — is recorded in s11 as the degrade option; v1 ships the faithful Scheduled-create.

## s12 · Register entry **R45** + back-propagation (write with the `S-mr-1` PR)

The convened review's **minutes** (auto-compiled 9.3.2 inputs *as-of* + the 9.3.3 decisions/outputs) **freeze into the version snapshot at submit** (the S-obj freeze recipe — rfc8785 JSON source + `metadata_snapshot.mgmt_review_minutes`, `no_controlled_rendition`); **release** files them as the immutable 9.3.3 retained record and flips 9.3 COVERED. **Sign-off** rides the standard `document.approve`/`document.release` (signed `meaning=approval`/`release` — **no new `SignatureMeaning`**, R2; **SoD** submitter≠approver≠releaser); `attendees` jsonb is the informational roster (a dedicated Top-Management routing is deferred). **Outputs** spawn `MR_ACTION` tasks at release (the `_materialize_stage` `task_type` seam — no engine/migration change); the **close gate mirrors `_audit_close_gate`** — the review closes only when every action's task is DONE (CAPA un-reserve + the DCR link are slice-2; the `review_output.spawned_capa_id`/`spawned_initiative_id` columns ship reserved-null). **Input compilation** runs under the caller's grants, **fail-closed per source → a named gap row, never a 403**, over the six already-shipped org-wide reads (objectives scorecard, audits, NCRs/complaints/CAPAs, KPI readings, compliance-checklist+overdue+drift); the four sourceless inputs + risk + improvement are honest gap rows. **Cadence** is a `system_config.mgmt_review_cadence_months` INTEGER (coded default 12; org-config later) driving a daily Beat sweep that mints the next Scheduled review + `MR_INPUT` task (mirroring `sweep_reviews`). **Permissions** ride the already-seeded SYSTEM `mgmtReview.read`/`create`/`record_outputs` (granted to QMS Owner in 0004) — **no new permission key, catalog stays 100, no R38 change**; `record_outputs` is non-signing (`sig_hook=False`, R43) with its `sod_sensitive=True` flag documentary-only (no engine path). **Audit** is additive `MGMT_REVIEW_*` event types on `object_type='document'` (R39 — no new `audit_object_type`).
