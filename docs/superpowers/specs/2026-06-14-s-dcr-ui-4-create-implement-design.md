# S-dcr-ui-4 — CREATE-implement + the CREATE resulting-doc deep-link (clause §10→§7.5) — design

- **Date:** 2026-06-14
- **Slice:** the LAST residual of the DCR-UI track (residual C). The SPA can drive a REVISE/RETIRE DCR end-to-end (ui-2b) and *see* a REVISE diff (ui-3), but a **CREATE** DCR is a dead-end in two places: its "Resulting version" field shows a bare id with no deep-link, and its Approved→Implement step is hard-excluded from the cockpit. This slice closes both. **After this, the DCR domain is fully complete in the SPA.**
- **Status:** owner-approved design (brainstorm 2026-06-14; F1+F2 settled WITH the owner after a 7-reader code-verification fan-out — all 7 probes `confirmed`).
- **Scope decision — ONE small backend touch + FE.** A detail-only `resulting_document_id` enrichment on `GET /dcrs/{id}` (migration-free — derived from an existing FK) + ONE contract field on the `Dcr` response schema; the rest is front-end. **No migration, no permission key, no new endpoint.** Gates: `/check-api` + `/check-contracts` + `/check-web` (head stays `0051`).
- **Doc grounding:** doc-05 (the change-control loop) · doc-15 §8.7 (the DCR API) · the ui-2b cockpit + `_dcr_capabilities` precedent · the web SPA testing rules (`.claude/rules/engineering-patterns.md` "Web SPA testing").

## s0 · As-built anchors (VERIFIED 2026-06-14 — pin to these, NOT narrative)
- **`_dcr` serializer** ([api/dcr.py:119‑138]) — emits `resulting_version_id` (133), `target_document_id` (123), `change_type` (124), `state`, etc. **Does NOT emit `resulting_document_id`.** ONE shared serializer (no detail-vs-list split); sync, takes a `dcr` row.
- **`_dcr_capabilities`** ([api/dcr.py:239‑259]) — detail-only, attached in `get_dcr_endpoint` ([api/dcr.py:379‑393], `out["capabilities"] = await _dcr_capabilities(...)` at :392); the list endpoint ([api/dcr.py:356‑376]) does NOT attach it. Scope resolves from `target_document_id`; for CREATE (`target_document_id is None`) it falls to the SYSTEM fallback, so `capabilities.implement` for a CREATE DCR = the `changeRequest.implement` answer ONLY (the ui-2b `_underlying_control_target` returns `None` for CREATE).
- **No top-level `GET /versions/{id}`** — all 8 version reads are nested under `/documents/{document_id}/versions/...` ([api/documents.py:1617‑1798]), gated `document.read_draft` with scope resolved from `document_id`. `_load_version` ([:1638‑1644]) enforces `version.document_id == document_id`. So the SPA cannot resolve a bare `resulting_version_id → document_id` without server help → the deep-link is NOT FE-only resolvable.
- **`document_version.document_id`** ([db/models/document_version.py:62‑66]) — a **non-nullable FK** to `documented_information.id`. So `resulting_document_id` is derivable at read with **no schema change**.
- **CREATE-implement** ([services/dcr/service.py:806‑896]) — `_resolve_implement_version` for CREATE **requires** a client-supplied `resulting_version_id` (422 if absent), fetches the `DocumentVersion`, validates it belongs to an in-org `kind=DOCUMENT` doc; `implement_dcr` sets `version.dcr_id = dcr.id`, `dcr.resulting_version_id = version.id`, and the implement **releases** that Approved version (the SERIALIZABLE cutover; SoD-2 on the new doc's `document.release`). REVISE auto-resolves the version from `target_document_id`; RETIRE leaves `resulting_version_id` NULL. The endpoint `POST /dcrs/{id}/implement` ([api/dcr.py:459‑492]) already accepts `DcrImplement.resulting_version_id`.
- **`DcrImplement` request contract** ([packages/contracts/openapi.yaml:6006‑6021]) — **already** documents `resulting_version_id` ("CREATE only … required for CREATE"). **No request-contract change needed.**
- **`Dcr` response contract** ([packages/contracts/openapi.yaml:6797‑6846]) — `additionalProperties: false`; has `resulting_version_id` (uuid|null) but **no `resulting_document_id`**. Adding the field is a load-bearing contract change (must land in `openapi.yaml`).
- **FE `DcrImplementBody`** ([apps/web/src/lib/types.ts:1470‑1471]) — **already** has `resulting_version_id?: string | null` ("CREATE only (deferred in the SPA)"). FE `Dcr` type ([:1395]) has `resulting_version_id` but **no `resulting_document_id`**.
- **FE drawer** ([apps/web/src/features/dcr/DcrDrawer.tsx:114‑137]) — "Resulting version" field: for REVISE/RETIRE (`target_document_id` set) renders "View document" → `/documents/{target_document_id}` (+ "View visual diff" → `/dcrs/{id}/diff` for REVISE); for CREATE renders bare `resulting_version_id.slice(0,8)… (new document)` Text, no link (comment at :116‑118 documents the limitation).
- **FE cockpit** ([apps/web/src/features/dcr/DcrAdvancePanel.tsx:31‑81]) — `canImplement = caps?.implement === true && state==="Approved" && change_type !== "CREATE"`; CREATE@Approved shows a calm note "New-document change requests are implemented from the document workspace." (:56‑60). `ImplementDcrModal` ([ImplementDcrModal.tsx]) is mounted only for `change_type !== "CREATE"` (REVISE `{}` / RETIRE `{force_retire, override_justification}`).
- **Existing FE hooks reused:** `useDocuments(filter, ...)` (the ui-2a target picker used `{current_state:"Effective"}` as a searchable Select); `useDocumentVersions(docId, {retry?})` (ui-3, widened with optional retry); `useImplementDcr(id)` ([features/dcr/mutations.ts:113‑121], `onSettled` invalidate, never optimistic); `usePermissions().can` / `dcr.capabilities`.

## s3 · Error handling, gating & data flow
- **Submit-and-show (the load-bearing gate honesty).** For CREATE, `dcr.capabilities?.implement` reflects only `changeRequest.implement` (the backend can't know the picked doc's `document.release`/SoD-2 scope until the user selects). So: show the Implement affordance on `capabilities.implement === true && state === "Approved"`, and **surface the real server gates calmly on submit** — `document.release` 403 (no release right on the picked doc), the SoD-2 `release_author_conflict`-style 409 (implementer == the new doc's author), the obsoletion/cutover 409s — rendered verbatim in the modal (the ui-2b `CloseAction` submit-and-show precedent). The modal does NOT pre-gate on the picked doc.
- **Authz (deep-link).** `resulting_document_id` is a bare UUID; reading the target doc is separately `document.read`-gated and the doc page calm-degrades — exactly as `target_document_id` already behaves for REVISE/RETIRE. Exposing the id leaks no more than the existing `target_document_id`.
- **Non-optimistic everywhere** — `useImplementDcr` is `onSettled`-invalidate (a 409 self-heals the drawer to the true state behind the calm error).
- **Candidate-set caveat (named, v1.x):** the picker lists ALL Approved documents — no stored DCR→doc link exists before implement, so the user asserts the linkage (they authored it). A tighter candidate set (e.g., docs created in the DCR's window) is a later refinement, not this slice.
